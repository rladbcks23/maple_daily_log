# 데이터 저장 기준

메이플 숙제알리미의 1차 목표는 데이터 분석이나 리포트가 아니라, 놓친 숙제와 이벤트를 알려주는 것이다.
따라서 처음부터 별도 데이터베이스를 크게 설계하지 않는다.

## 1. 1차 방향

- 서버는 Nexon OpenAPI를 호출하고 결과를 앱에 전달한다.
- 서버는 리포트, 성장 기록, 장비 비교용 데이터를 저장하지 않는다.
- 스케줄러/공지 API 응답은 알림 판단에 필요한 범위에서만 사용한다.
- 가능하면 1차 구현은 DB 없이 시작한다.

## 2. DB 없이 가능한 기능

- 오늘 접속 여부 확인
- 일일 콘텐츠 미완료 여부 확인
- 주간 콘텐츠 미완료 여부 확인
- 보스 콘텐츠 미완료 여부 확인
- 진행 중 이벤트 목록 확인
- 공지사항, 캐시샵 공지, 업데이트 목록 확인
- 메이플 런처 종료 시점에 API를 호출해 남은 일일 이벤트 확인

## 3. 저장이 필요해질 수 있는 정보

아래 기능이 필요해지는 시점에만 저장소를 추가한다.

- 이미 보여준 공지인지 확인
- 이미 알림을 띄운 항목인지 확인
- 사용자가 끄거나 켠 알림 설정
- 사용자가 중요하게 보는 캐릭터 또는 콘텐츠 설정
- 사용자가 알림 대상으로 선택한 캐릭터
- 선택한 캐릭터의 `ocid`
- 알림 시간 설정

## 4. 저장소 후보

1차에서는 서버 DB보다 앱 로컬 저장을 우선 검토한다.

- 앱 로컬 저장:
  - 알림 설정
  - 알림 대상으로 선택한 캐릭터
  - 선택한 캐릭터의 `ocid`
  - 마지막으로 본 공지 식별자
  - 마지막 알림 시간
- 서버 DB:
  - 여러 기기 동기화가 필요할 때
  - 서버가 독립적으로 예약 수집과 알림 상태를 관리해야 할 때

## 5. 보류한 기존 DB 설계

기존 `schema.sql`은 캐릭터 스냅샷, 플레이 시간, 리포트, 강화 이력 저장을 위한 구조였다.
새 기획에서는 해당 기능을 만들지 않으므로 삭제한다.
