메일기 현재 구상 간단 메모
작성일: 2026-07-08

1. 전체 구조

- 서버: 데이터 수집 / 저장 / 분석 / 리포트 작성
- 로컬 앱: UI/UX / 내 컴 게임 실행 감지 / 수기 입력 / 간단 계산
- 서버는 개인용 수집기 느낌
- 앱은 사용자가 보는 일기장 느낌 --> 나중에 따로 기획 예정


2. 서버 역할

- Nexon OpenAPI 호출
- 캐릭터 목록 동기화
- API 데이터 기반 기록 저장
- 누락분 백필
- 현재 상태 조회 및 이전 기록과 비교
- 변경점 계산
- 일간/주간/월간 리포트 작성
- 공식 가격표 기반 재화 계산
- 외부 PC 또는 미기록 플레이 흔적 감지
- 로컬 앱에 데이터 전달


3. 로컬 앱 역할

- 실제 화면/UI
- 내 컴에서 메이플 실행 감지
- 게임 시작/종료 시간 기록
- 내 PC 플레이타임 계산
- 수동 새로고침
- 수기 플레이타임 입력
- 미입력 플레이타임 알림
- 캐릭터 태그 관리
- 서버 데이터 보기


4. 데이터 저장

- DB는 Supabase를 사용
- 1차에서는 Supabase를 인증 서비스가 아니라 DB 용도로만 사용
- 서버는 Spring Boot에서 Supabase DB에 연결
- WebSocket은 저장소가 아니라 실시간 알림/갱신용
- 1차는 REST API만으로 충분
- API로 조회한 현재 상태 자체를 계속 스냅샷으로 누적하지는 않음
- 현재 상태는 필요할 때 API로 다시 불러오고, DB에는 캐릭터 기본 정보 / 변화 기록 / 리포트 재료 / 리포트 결과를 저장


5. 보안

- 서버는 다른 사람 접근 불가하게
- API Key/token은 코드에 박지 않기
- .env 또는 환경변수 사용
- .env, DB, 로그는 Git에 올리지 않기
- 관리자 토큰 필요
- 외부 포트 최소화
- 가능하면 HTTPS


6. API 호출 타이밍

- 앱 실행 시
- 게임 실행 감지 시
- 게임 종료 감지 시
- 수동 새로고침 시
- 새벽 또는 다음날 정산 시
- 외부 이벤트 호출 사이트를 통해 서버 슬립 방지 및 정기 트리거 가능

실행 중 주기 호출은 1차에서는 안 해도 됨.
필요하면 나중에 실시간 모드로 추가.


7. 앱 실행 시 할 일

- 누락분 백필
- 최근 강화 로그 회수

- 주요 캐릭터 현재 상태 저장
- 미입력 플레이타임 알림

- 시작 시간 기록
- 플레이 전 스냅샷 저장
- 기본 정보/경험치
- 스탯/전투력
- 장비
- 심볼
- 헥사/헥사 스탯
- V매트릭스
- 어빌리티
- 하이퍼스탯
- 스케줄러


9. 게임 종료 감지 시 할 일

- 종료 시간 기록
- 플레이 후 최종 스냅샷 저장
- 시작 전/종료 후 차이 계산
- 강화 로그 회수
- 스케줄러 최종 상태 확인


10. 자동 기록 가능

- 경험치 현재량
- 경험치 퍼센트
- 경험치 증가량
- 레벨업
- 전투력
- 세부 스탯
- 어빌리티
- 하이퍼스탯
- 착용 장비
- 캐시 장비
- 심볼
- 5차 코어
- 6차 헥사 코어
- 6차 헥사 스탯
- 유니온
- 길드
- 스케줄러 일일/주간 보스 상태
- 스타포스 로그
- 큐브 로그
- 잠재 재설정 로그


11. 자동 기록 어려움

- 보유 메소
- 인벤토리 전체
- 창고
- 경매장 기록
- 거래 기록
- 보유 소비 아이템
- 정확한 로그인/로그아웃 시간
- 다른 PC에서의 정확한 플레이타임
- 추옵 재설정 횟수
- 경험치 쿠폰 사용/획득 수량


12. 강화/재화

- 스타포스: 로그 있음, 실제 메소 필드는 없음, 공식 가격/할인율로 계산
- 잠재/에디셔널: 로그 있음, 실제 메소 필드는 없음, 공식 가격표로 계산
- 큐브: 사용 로그/큐브 타입/횟수는 가능, 구매비용/획득경로는 모름
- 추옵: 로그 없음, 최종 변경만 감지
- 헥사: 레벨 변화 + 공식 비용표로 솔 에르다/조각 계산 가능
- 가격표/비용표는 코드에 직접 박지 않고 별도 설정 파일로 분리
- .env처럼 환경별로 분리해서 관리하되, 단순 비밀값이 아니므로 구조화 파일(JSON/YAML 등) 후보


13. 플레이타임

- 내 PC: 로컬 앱이 게임 실행/종료 감지해서 계산
- 다른 PC: 자동 계산 불가, 수기 입력 필요
- 시간은 분 단위로 저장
- 게임을 켠 상태로 날짜가 넘어가도, 해당 세션은 게임을 켠 날의 기록으로 저장
- 스케줄러 API로 해당 날짜 접속 가능성은 간접 감지 가능
- 다른 PC에서 언제 껐는지는 알 수 없음
- 캐릭터 변경은 직접 감지 불가, 활동 로그로 추정만 가능


14. 미기록 플레이 처리

조건:
- 활동 로그/변경점 있음
- 로컬 게임 실행 세션 없음
- 수기 플레이타임 없음

결과:
- 앱 실행 시 알림
- "이 PC에서 기록되지 않은 플레이가 있어요. 플레이 시간을 입력할까요?"
- 입력한 시간 내에 있는 로그들을 자동으로 그날 리포트로 저장

상태값:
- missing: 활동 흔적 있음, 플레이타임 없음
- pending: 알림 표시한 적 있음
- filled: 사용자가 입력함
- ignored: 사용자가 무시함

16. 캐릭터 태그

- main: 본캐
- active_sub: 자주 키우는 부캐
- boss_mule: 보스돌이
- union: 유니온 캐릭
- storage: 창고/거의 안 쓰는 캐릭
- ignore: 기록 제외

태그마다 수집 범위와 리포트 포함 여부를 다르게 설정.
사실상 밑 3개는 데이터 수집 안함

17. 데이터 정확도 라벨

- API_REPORTED: API가 직접 준 원본 로그
- SNAPSHOT_DIFF: 전후 스냅샷 비교
- RULE_CALCULATED: 공식 가격표/규칙 계산
- ESTIMATED: 추정
- MANUAL: 사용자 직접 입력


18. 리포트

리포트 공통:
- 테이블은 report 하나로 시작
- report_type으로 daily / weekly / monthly 구분
- 1차는 daily만 사용하고, weekly / monthly는 이후 추가

일일 리포트:
- 플레이타임
- 경험치 증가량 ( + 레벨업)
- 전투력 변화
- 장비 변경점( + 헥사/심볼 성장)
- 강화 기록
- 재화 사용 계산
- 스케줄러/보스 완료

주간 리포트:
- 총 플레이타임
- 총 경험치 증가
- 전투력 변화
- 강화 횟수
- 재화 사용 추정
- 주간 보스 현황

월간 리포트:
- 월간 성장량
- 총 플레이타임
- 총 강화 횟수
- 총 재화 사용 추정
- 레벨/전투력 추이
- 캐릭터별 성장 비교

20. 현재 결론

- 서버는 조용히 API를 수집하고 저장/분석한다.
- 로컬 앱은 UI, 내 PC 실행 감지, 수기 보정을 담당한다.
- API 호출은 앱 실행, 게임 실행, 게임 종료, 수동 새로고침, 정산 시점에 한다.
- 실행 중 주기 호출은 1차에서는 제외한다.
- 정확한 값과 추정값은 반드시 구분한다.
- 서버 위치는 Render 무료 배포로 한다.
- 서버 기술은 Spring Boot로 한다.
- 앱 기술은 Flutter로 한다.
- DB는 Supabase를 DB 용도로만 사용한다.
- 인증은 HTTPS + 관리자 토큰으로 한다.
- 1차 MVP는 나 혼자 사용하는 메이플 기록용으로 제한한다.
- 일일 리포트는 한국 시간 기준 하루로 묶는다.
- 주간 리포트는 메이플 주간 초기화 기준으로 묶는다.
- 원본 스냅샷 시간은 KST로 표시한다.
- 현재 상태 스냅샷을 계속 저장하는 방식은 사용하지 않는다.
- 변화가 생긴 기록과 리포트에 들어갈 누적 데이터 위주로 저장한다.



21. 1차 MVP 범위

- 캐릭터 목록 동기화
- 기본 정보 / 레벨 / 경험치 저장
- 전투력 / 스탯 저장
- 장비 저장
- 심볼 저장
- 헥사 / 헥사 스탯 저장
- 스타포스 로그 저장
- 큐브 로그 저장
- 잠재 재설정 로그 저장
- 일일 리포트 작성

1차 이후 추가 후보:
- 보스/콘텐츠 체크
- 주간 리포트
- 월간 리포트
- 캐릭터별 성장 비교
- 더 세밀한 재화 계산


22. DB 구조 초안

기본 방향:
- 캐릭터 기본 정보는 별도 테이블로 관리
- API 현재 상태는 필요할 때 조회
- 업데이트마다 늘어나는 데이터는 리포트 재료로 누적
- 플레이타임은 분 단위 총합으로 저장
- 리포트는 report 단일 테이블에 daily / weekly / monthly 타입으로 구분

1차 테이블 후보:
- characters: 캐릭터 기본 정보, ocid, 월드, 직업, 태그, 기록 제외 여부
- character_records: 레벨, 경험치, 전투력, 주요 스탯 등 업데이트마다 저장되는 성장 기록
- equipment_records: 장비 변경 기록
- symbol_records: 심볼 성장 기록
- hexamatrix_records: 헥사 / 헥사 스탯 성장 기록
- starforce_events: 스타포스 로그
- cube_events: 큐브 로그
- potential_events: 잠재 재설정 로그
- play_time_records: 날짜별 총 플레이타임, 분 단위 저장, 수동 입력 여부
- reports: report_type으로 daily / weekly / monthly 구분, 리포트 결과 JSON 및 요약값 저장
